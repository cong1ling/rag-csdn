# 前端具体实现文档

## 1. 项目定位

`rag-csdn-front` 是当前 CSDN 文章 RAG 项目的前端工作台，用于完成以下核心业务：

- 用户注册、登录、登出
- CSDN 文章导入
- 已导入文章管理
- 会话创建与删除
- 基于文章知识库的流式问答

项目采用 `Vue 3 + Vite + Pinia + Vue Router + Element Plus` 实现，强调单页应用体验、快速联调和流式问答交互。

## 2. 目录结构

核心结构如下：

```text
rag-csdn-front/
├─ src/
│  ├─ api/                 # 接口封装
│  ├─ components/          # 通用组件
│  ├─ config/              # 环境配置
│  ├─ constants/           # 常量与状态映射
│  ├─ mock/                # 本地开发模式模拟服务
│  ├─ router/              # 路由配置与守卫
│  ├─ stores/              # Pinia 状态
│  ├─ styles/              # 全局主题样式
│  ├─ utils/               # 工具函数
│  ├─ views/               # 页面级组件
│  ├─ App.vue              # 根组件
│  └─ main.js              # 应用入口
├─ index.html
├─ vite.config.js
├─ package.json
└─ .env.example
```

## 3. 启动链路

前端入口位于 `src/main.js`，启动流程如下：

1. 创建 Vue 应用实例
2. 注册 Pinia
3. 注册 Vue Router
4. 引入全局主题样式 `src/styles/theme.css`
5. 挂载到 `#app`

这意味着前端是标准 SPA 架构，页面切换完全依赖路由控制。

## 4. 环境与代理设计

环境变量由 `src/config/env.js` 读取：

- `VITE_API_BASE_URL`：接口基础路径，默认 `/api`
- `VITE_PROXY_TARGET`：Vite 开发代理目标，默认 `http://localhost:8080`

`vite.config.js` 中定义了开发代理规则：

- 浏览器请求 `/api/*`
- Vite 自动转发到后端服务

这样做有两个直接收益：

- 本地开发时避免手写完整后端域名
- 登录接口、列表接口、流式接口统一走相同前缀

## 5. 路由设计

路由位于 `src/router/index.js`。

### 5.1 路由表

- `/` 主页
- `/login` 登录页，仅游客可访问
- `/register` 注册页，仅游客可访问
- `/import` 文章导入页，需要登录
- `/articles` 文章列表页，需要登录
- `/videos` 兼容跳转到 `/articles`
- `/sessions` 会话列表页，需要登录
- `/chat/:sessionId` 对话页，需要登录

### 5.2 路由守卫

前置守卫负责以下工作：

1. 首次进入页面时调用 `authStore.fetchCurrentUser()`
2. 如果目标页要求登录但当前未登录，则跳转到 `/login`
3. 如果目标页仅游客可访问但用户已登录，则跳转到 `/articles`
4. 根据页面 `meta.title` 设置浏览器标题

这套逻辑把“登录态初始化”和“页面访问控制”集中在路由层处理，页面本身不需要重复判断。

## 6. 状态管理

状态管理主要在 `src/stores/auth.js`。

当前只抽离了认证状态，这个选择是合理的，因为：

- 登录用户信息会跨页面共享
- 路由守卫依赖统一认证状态
- 登出后需要全局恢复未登录状态

### 6.1 认证状态字段

- `user`：当前登录用户
- `initialized`：是否完成首次鉴权初始化
- `loading`：认证相关请求是否进行中

### 6.2 计算属性

- `isAuthenticated`：是否已登录
- `isDeveloperMode`：是否启用开发模式
- `canUseDeveloperEntry`：是否展示开发入口

### 6.3 核心动作

- `fetchCurrentUser()`：初始化用户态
- `register(payload)`：注册
- `login(payload)`：登录
- `enterDeveloperMode(payload)`：进入开发模式
- `logout()`：退出登录

## 7. 开发模式设计

`src/utils/dev-mode.js` 实现了一个隐藏的开发模式。

核心特点：

- 通过 `ENABLE_DEVELOPER_ENTRY` 控制是否开放
- 当前代码中该开关为 `false`
- 若开启后，可使用固定测试账号进入本地开发模式
- 开发模式标记保存在 `sessionStorage`

当开发模式启用时，部分接口不会访问真实后端，而是走 `src/mock/dev-server.js`。

当前接入本地模拟的模块包括：

- 文章导入与文章管理
- 会话管理
- 消息列表与流式问答

认证模块中的“获取当前用户”也支持直接返回本地模拟用户。

这个设计适合前端独立联调、演示和局部调试，但默认关闭，避免误入测试逻辑。

## 8. 接口层实现

接口封装集中在 `src/api/`。

### 8.1 通用 HTTP 客户端

`src/api/http.js` 基于 Axios 创建实例，主要配置如下：

- `baseURL` 来自 `apiBaseUrl`
- `timeout: 20000`

并通过拦截器实现：

- 从 `localStorage` 读取 JWT 并自动附加 `Authorization: Bearer <token>`
- 请求日志记录
- 响应统一拆包
- 非 `code === 200` 统一转为异常
- 未登录错误自动跳回登录页

说明：当前前后端约定响应结构是“外层统一 Result 包装”，因此前端响应拦截器会直接返回 `payload.data`。

### 8.2 认证接口

`src/api/auth.js`

- `register`
- `login`
- `logout`
- `current`

### 8.3 文章接口

`src/api/articles.js`

- `importArticle`
- `rebuild`
- `list`
- `detail`
- `remove`

### 8.4 会话接口

`src/api/sessions.js`

- `create`
- `list`
- `detail`
- `remove`

### 8.5 消息接口

`src/api/messages.js`

- `list(sessionId)`
- `stream(sessionId, payload, handlers, signal)`

其中 `stream` 没有复用 Axios，而是直接使用浏览器 `fetch`，原因是它需要消费服务端的流式返回。

## 9. SSE 流式消息实现

流式问答是该前端最关键的交互能力之一。

### 9.1 实现位置

- 接口入口：`src/api/messages.js`
- SSE 解析：`src/utils/sse.js`
- 页面消费：`src/views/ChatView.vue`

### 9.2 实现方式

前端向：

```text
/sessions/{sessionId}/messages/stream
```

发起 `POST` 请求，并通过 `fetch` 读取流式响应体。

`consumeSseStream()` 的职责包括：

1. 使用 `ReadableStream.getReader()` 按块读取响应
2. 以 `\n\n` 切分 SSE 事件
3. 解析 `event:` 和 `data:` 字段
4. 优先把 `data` 当作 JSON 解析
5. 根据事件类型分发到 `start`、`content`、`end`、`error` 等处理器
6. 汇总完整回答文本，结束时回传 `fullContent`

### 9.3 页面侧消费策略

`ChatView.vue` 采用了“先插入占位消息，再增量填充内容”的方案：

1. 用户点击发送
2. 立即在本地消息列表插入一条用户消息和一条空的助手消息
3. 服务端持续返回增量文本
4. 前端使用 `requestAnimationFrame` 合并多次字符增量，减少频繁 DOM 更新
5. 流结束后重新拉取一次消息列表，确保最终状态与服务端一致

这个方案兼顾了：

- 首屏反馈速度
- 流式输出观感
- 与最终服务端落库结果对齐

### 9.4 中断逻辑

页面使用 `AbortController` 管理当前流：

- 用户点击“停止生成”时调用 `abort()`
- 页面卸载时自动中止
- 中止后根据是否已经开始、是否已经完成，决定保留或回滚本地占位消息

## 10. 页面实现说明

### 10.1 登录页与注册页

职责：

- 收集用户名和密码
- 调用认证接口
- 注册成功后跳转登录或业务页
- 登录成功后跳转到原始目标页或文章列表页

### 10.2 导入页 `ImportView.vue`

核心功能：

- 输入完整 CSDN 文章 URL
- 调用 `articlesApi.importArticle()`

实现细节：

- 导入结果会直接展示文章标识、标题、来源链接和导入时间
- 成功后可直接跳转到文章列表页

### 10.3 文章列表页 `ArticlesView.vue`

核心功能：

- 展示文章统计
- 关键字筛选标题和文章标识
- 查看文章详情
- 对历史文章重建索引
- 删除文章
- 从单篇文章直接创建会话并跳转到聊天页

实现细节：

- `readyCount` 和 `failedCount` 通过计算属性统计
- “重建索引”通过弹窗确认重建目标，调用 `articlesApi.rebuild()`
- 删除前弹出二次确认框
- 状态展示依赖 `VIDEO_STATUS_META`

### 10.4 会话列表页 `SessionsView.vue`

核心功能：

- 加载会话列表
- 加载可选文章列表
- 新建单文章会话或全库会话
- 删除会话
- 进入聊天页

实现细节：

- 页面初始化时并行请求会话列表和文章列表
- 新建会话弹窗组件为 `SessionDialog.vue`
- 仅状态为 `SUCCESS` 的文章可用于创建单文章会话

### 10.5 聊天页 `ChatView.vue`

核心功能：

- 加载会话详情
- 加载历史消息
- 发送问题
- 接收流式回答
- 停止生成

实现细节：

- 通过路由参数 `sessionId` 识别当前会话
- 地址非法时自动跳回会话列表
- 消息渲染时为每条记录补充 `localKey`，避免临时消息没有服务端 ID 时无法稳定渲染

## 11. 通用组件设计

### 11.1 `AppShell.vue`

作用：

- 统一左侧导航
- 展示当前用户信息
- 提供页面头部区域
- 统一退出登录入口

这是整个业务后台风格的布局壳组件。

### 11.2 `EmptyState.vue`

作用：

- 空数据场景占位展示

适用于：

- 没有视频
- 没有会话
- 没有消息

### 11.3 `StatusPill.vue`

作用：

- 统一展示状态标签

适用于：

- 视频状态
- 会话类型
- 消息角色
- 聊天生成状态

### 11.4 `SessionDialog.vue`

作用：

- 创建会话的弹窗表单

## 12. 常量与工具函数

### 12.1 常量

`src/constants/options.js` 中维护了业务枚举到展示文案的映射，例如：

- 视频状态
- 消息角色

这样做可以把“后端状态值”和“前端显示文案”解耦。

### 12.2 工具函数

`src/utils/` 中主要包含：

- `error.js`：异常归一化、错误提示
- `format.js`：时间格式化、文本裁剪
- `logger.js`：日志输出与敏感数据脱敏
- `sse.js`：SSE 解析与消费
- `dev-mode.js`：开发模式控制

## 13. 样式层设计

全局主题样式位于：

```text
src/styles/theme.css
```

当前项目的样式策略是：

- 页面级布局统一走 `AppShell`
- 卡片、状态块、工具栏等通过通用类复用
- 主要视觉层次通过 `surface`、`surface-strong`、`card-section` 等类名表达

这种方式的优点是结构稳定、开发效率高，适合中后台型工作台界面。

## 14. 前后端协作约定

从当前前端实现可以归纳出以下后端约定：

- 所有普通接口返回统一结果结构，成功码为 `200`
- 未登录时返回特定错误码，前端据此自动跳转登录页
- 聊天接口支持 SSE 风格的流式输出
- 会话、消息、视频均按当前登录用户隔离

如果后端变更这些约定，前端需要同步调整：

- `http.js` 响应拦截器
- `error-codes.js`
- `messages.js`
- `sse.js`

## 15. 当前实现特征总结

这个前端实现有几个明显特征：

- 路由守卫统一处理登录态和页面准入
- API 层与页面层职责分离清晰
- 流式问答单独走 `fetch`，没有强行复用 Axios
- 支持隐藏开发模式，便于前端脱离后端进行局部演示
- 页面围绕“导入视频 -> 管理视频 -> 创建会话 -> 流式问答”这条主链路展开

如果后续继续扩展，优先建议的方向是：

- 增加接口类型定义或 DTO 约束
- 补充前端单元测试与页面集成测试
- 对聊天页补充更细粒度的流状态展示
- 为导入、删除、会话创建增加更完整的操作反馈
